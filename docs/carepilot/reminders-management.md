# CarePilot Reminders Management

## Purpose
CarePilot Reminders is the operational control center for reminder executions generated by campaigns and trigger services.

It provides:
- reminder filtering and status grouping
- timeline visibility (attempts + provider events)
- retry/resend actions for failed executions
- cancel/suppress/reschedule actions for queued and retrying reminders

## Status Meanings
- `QUEUED` / `PROCESSING`: pending delivery pipeline states
- `RETRY_SCHEDULED`: retry engine scheduled next attempt
- `FAILED` / `DEAD_LETTER`: execution send failed
- delivery lifecycle (`SENT`, `DELIVERED`, `READ`, `FAILED`, `BOUNCED`, `UNDELIVERED`, `SKIPPED`) comes from send-attempt + webhook sync

## Operational Actions
Implemented:
- `Retry`
- `Resend` (aliases retry flow)
- `Cancel`
- `Suppress`
- `Reschedule`

Action rules:
- `Cancel`: only `QUEUED`, `PROCESSING`, `RETRY_SCHEDULED`
- `Suppress`: only `QUEUED`, `PROCESSING`, `RETRY_SCHEDULED`
- `Reschedule`: only `QUEUED`, `PROCESSING`, `RETRY_SCHEDULED` and `newScheduledAt` must be future
- delivered/read/succeeded reminders are rejected for these actions

## RBAC
- `CLINIC_ADMIN`: read + mutate (retry/resend/cancel/suppress/reschedule)
- `AUDITOR`: read-only
- `PLATFORM_ADMIN` / `PLATFORM_TENANT_SUPPORT`: tenant-scoped access
- `DOCTOR`, `BILLING_USER`, `RECEPTIONIST`: no reminders management access by default policy

## Limitations
- Timeline actor identity is not yet persisted for reminder mutation actions.
- Page uses reminders query API + existing timeline infrastructure; no duplicate execution logic.
